## 1. Назначение системы

Dental CRM — внутренняя система управления стоматологической клиникой. Система охватывает полный операционный цикл:

- Запись пациентов по телефону и онлайн
- Расписание врачей с визуальным календарём
- Ведение медицинских карт (зубная формула, рентген, план лечения)
- Учёт оплат и долгов
- Автоматические SMS-напоминания пациентам
- Аналитика и отчётность для администрации

Система используется тремя категориями пользователей: администраторами клиники, врачами и пациентами.

## 4. Модули системы

---

### М1 — Core (✅ Сделано)

Базовые сущности и аутентификация.

#### 4.1 Аутентификация

- Вход по `username` + пароль → JWT access + refresh токен
- JWT payload: `role`, `user_id`, `username`
- Auto-refresh на 401 через Axios interceptor
- Logout → очистка localStorage
- Роут-гард `ProtectedRoute` с проверкой роли
- Сброс пароля через OTP-код (SMS-заглушка, реальный провайдер — М5)

**API:**
```
POST /api/v1/token/           — вход
POST /api/v1/token/refresh/   — обновление access
POST /api/v1/users/forgot-password/  — запросить OTP
POST /api/v1/users/reset-password/   — сменить пароль по OTP
```

#### 4.2 Пациенты (Client)

Полная карточка пациента.

**Поля:** ФИО, ИИН (12 цифр), телефон, пол, дата рождения, адрес, источник обращения, LTV, привязанный врач.

**API:**
```
GET    /api/v1/clients/         — список (поиск: ФИО, ИИН, телефон)
POST   /api/v1/clients/         — создание (только admin)
GET    /api/v1/clients/<id>/    — карточка
PATCH  /api/v1/clients/<id>/    — редактирование
DELETE /api/v1/clients/<id>/    — удаление (только admin)
```

#### 4.3 Врачи и услуги

- **Doctor** — ФИО, специализация, услуги, описание, расписание
- **Specialization** — название
- **Service** — название, специализация, стоимость по умолчанию
- **Assistant** — ФИО, привязка к услуге

#### 4.4 Записи на приём (Record)

Основная транзакционная сущность.

**Поля:** пациент, врач, ассистент, услуга, зуб, стоимость, скидка, итог, дата/время начала и конца, кабинет, тип оплаты, статус оплаты, статус приёма, заметки врача.

**Типы услуг (service_type):**

| Код | Описание |
|---|---|
| `exam` | Осмотр / консультация |
| `treatment` | Лечение (пломба, удаление и т.д.) |
| `cleaning` | Профессиональная чистка |
| `whitening` | Отбеливание |
| `prosthetics` | Протезирование / имплант |
| `orthodontics` | Ортодонтия (брекеты, трейнеры) |
| `surgery` | Хирургия (удаление, имплантация) |
| `pediatric` | Детский приём |
| `xray` | Рентген |
| `other` | Прочее |

#### 4.5 Медицинская карта

- **MedicalNote** — анамнез, аллергии, общие заметки
- **ToothRecord** — состояние каждого зуба (зубная формула 32 зуба)
- **TreatmentPlanItem** — позиция плана лечения по зубу
- **PatientFile** — файлы (рентген, фото, документы)

### М5 — Уведомления / SMS (🔄 Запланировано)

#### Автоматические напоминания о записях

- За 24 часа до приёма → SMS пациенту
- За 2 часа до приёма → SMS пациенту
- Ручная отправка напоминания из карточки записи

#### Реактивационные кампании (ключевая фича)

Автоматические сообщения пациентам с незавершённым планом лечения:

**Сценарии:**

| Триггер | Шаблон сообщения |
|---|---|
| Зуб в плане лечения → прошло 30 дней без новой записи | «[Имя], напоминаем: зуб №[N] требует лечения. Запишитесь: [ссылка]» |
| Установлена пломба → прошло 12 месяцев | «[Имя], рекомендуем профилактический осмотр. Пломба на зубе №[N] — прошёл год.» |
| Последний визит > 6 месяцев назад | «[Имя], вы не были у нас 6 месяцев. Приходите на чистку!» |
| Удаление нерва → 3 недели без повторной записи | «[Имя], после лечения зуба №[N] необходима контрольная коронка. Звоните: [телефон]» |

**Архитектура:**
- Celery Beat — периодические задачи (проверка раз в день в 10:00)
- Таблица `SMSLog`: запись_id, пациент_id, шаблон, статус, время
- Дедупликация: не отправлять одно сообщение дважды за 30 дней
- Настройки: включить/выключить каждый тип кампании в панели admin

**SMS-провайдер:** SMS.kz или SMSC.ru (выбирается при реализации).

**API:**
```
POST /api/v1/notifications/send/<record_id>/  — ручная отправка
GET  /api/v1/notifications/log/               — история
GET  /api/v1/notifications/campaigns/         — список кампаний
POST /api/v1/notifications/campaigns/<id>/toggle/ — вкл/выкл
```

### М8 — Перевод записи (🔮 Будущее)

Если врач не сможет принять пациента (болезнь, отпуск), администратор переводит запись другому врачу.

- Выбрать другого врача
- Выбрать новое время (с проверкой конфликтов)
- Автоматически уведомить пациента об изменении (SMS)
- Записать причину перевода
- Исходный врач видит перевод в своей истории

**Модель:**

```python
class RecordTransfer(models.Model):

    reason         = TextField(blank=True)
    transferred_by = ForeignKey(User, related_name='transfers_made')
    transferred_at = DateTimeField(auto_now_add=True)
```

### М10 — 3D Зубная схема (🔮 Будущее)

Интерактивная визуализация зубного ряда в карточке пациента.

#### Концепция

Врач в рамках приёма:
1. Видит схему всех 32 зубов пациента (SVG или Canvas)
2. Кликает на проблемный зуб
3. В модальном окне выбирает статус зуба и вводит комментарий
4. Статус сохраняется в медкарту

#### Статусы зубов

| Код | Значение | Цвет |
|---|---|---|
| `healthy` | Здоровый | Белый |
| `treated` | Пролечен | Зелёный |
| `filling` | Есть пломба | Жёлтый |
| `crown` | Коронка | Синий |
| `implant` | Имплант | Тёмно-синий |
| `needs_treatment` | Требует лечения | Оранжевый |
| `urgent` | Срочно (боль) | Красный |
| `extracted` | Удалён | Серый |
| `missing_congenital` | Отсутствует (врождённо) | Серый пунктир |

#### Модель

```python
class ToothRecord(models.Model):
```

### М11 — Рентген и диагнозы (🔮 Будущее)

После загрузки рентгена врач добавляет детальное описание.

#### Флоу

1. Администратор/врач загружает файл рентгена (jpg/png/dicom)
2. Врач открывает рентген в системе
3. Добавляет аннотации к рентгену: выбирает зуб, добавляет диагноз и план лечения
4. Система автоматически создаёт позицию в `TreatmentPlanItem`
5. Если пациент не может оплатить сейчас — отмечается флаг `deferred_payment` с заметкой
6. Запись попадает в триггеры реактивационных SMS (М5)

**Новые поля для TreatmentPlanItem:**
```python
class TreatmentPlanItem(models.Model):
    patient          = ForeignKey(Client, on_delete=CASCADE)
    tooth_num        = PositiveSmallIntegerField()
    diagnosis        = TextField()
    recommended_service = ForeignKey(Service, null=True)
    estimated_cost   = DecimalField(null=True)
    priority         = CharField(choices=['urgent','medium','low'])
    deferred_payment = BooleanField(default=False)  # пациент не может платить сейчас
    notes            = TextField(blank=True)
    completed        = BooleanField(default=False)
    completed_at     = DateTimeField(null=True)
    created_by       = ForeignKey(User, on_delete=SET_NULL, null=True)
    created_at       = DateTimeField(auto_now_add=True)
    linked_xray      = ForeignKey(PatientFile, null=True, blank=True)
```

#### Frontend

- В карточке пациента: вкладка «Рентген» → просмотр файлов с подписями
- Рядом с рентгеном — список аннотаций врача
- Кнопка «Добавить в план лечения» автоматически создаёт `TreatmentPlanItem`
- Пометка «отложено по финансам» — отображается в плане лечения для врача/admin

**API:**

## 5. Запись пациента — рабочий процесс администратора

### Сценарий «Звонок в ресепшн»

1. Пациент звонит → администратор открывает «Новая запись»
2. Поиск пациента по ФИО / телефону / ИИН:
   - Найден → открыть карточку
   - Не найден → создать нового прямо из формы записи
3. Выбрать врача → отображаются занятые слоты
4. Выбрать дату и время из доступных слотов
5. Выбрать услугу из списка (осмотр, лечение, чистка и т.д.)
6. Опционально: указать кабинет, ассистента, комментарий
7. Сохранить → запись попадает в расписание

### UX-требования к форме записи

- Автодополнение при поиске пациента (минимум 2 символа)
- Свободные слоты отображаются в виде сетки времени
- При выборе услуги подставляется стоимость по умолчанию (редактируемая)
- Если у пациента есть незавершённый план лечения — показывать уведомление

---

## 6. Рабочий процесс врача

### Приём пациента

1. Врач открывает расписание → видит свои записи на день
2. Выбирает запись пациента → открывается карточка с историей
3. Проводит осмотр:
   - Отмечает состояние зубов на схеме (М10)
   - Добавляет заметки к визиту
   - Загружает рентген (если делался)
   - Составляет план лечения с аннотациями к рентгену (М11)
4. После процедуры:
   - Меняет статус записи на «Завершён»
   - Администратор фиксирует оплату

### Финансовая прозрачность для врача

Врач видит (ограниченно):
- Сумму каждой своей записи
- Общую выручку за день / месяц (только по своим приёмам)
- Не видит: общую кассу клиники, данные других врачей

## 7. Медицинская карта пациента — структура

```
Пациент
├── Основная информация (ФИО, ИИН, телефон, адрес)
├── Медицинский анамнез
│   ├── Аллергии
│   ├── Хронические заболевания
│   └── Общие заметки
├── Зубная формула (32 зуба, М10)
│   └── Каждый зуб: статус, комментарий, история изменений
├── История посещений
│   ├── Дата, врач, услуга, стоимость, заметки врача
│   └── Прикреплённые файлы к визиту
├── Рентген и файлы
│   ├── Тип: рентген / фото / документ / before_after
│   └── Аннотации (М11)
├── План лечения
│   ├── По зубам: диагноз, рекомендуемая услуга, стоимость, приоритет
│   ├── Флаг «отложено по финансам»
│   └── Статус: выполнено / в процессе / ожидает
└── Финансовая история
    ├── Оплаченные записи
    └── Долги
```

## 8. Нефункциональные требования

| Требование | Значение |
|---|---|
| Время ответа API | < 300 мс для списков, < 100 мс для простых GET |
| Покрытие тестами | ≥ 80% на каждый app |
| Аутентификация | JWT на всех endpoints, кроме `/token/` и публичного `/booking/` |
| Безопасность файлов | Только через `/download/` с JWT |
| Пагинация | Page size 20 по умолчанию, настраиваемый |
| CORS | `localhost:5173` в dev, продакшн домен в prod |
| Секреты | `SECRET_KEY`, `DATABASE_URL`, `SMS_API_KEY` — только через env |
| Логи | `logging.getLogger(__name__)`, никаких `print()` |
| Throttling | OTP: 5 запросов/час. API auth: 100/час |
| Язык интерфейса | Русский (все лейблы, сообщения об ошибках, SMS-шаблоны) |
