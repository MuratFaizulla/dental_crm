## 1. Общие сведения

Dental CRM — внутренняя система управления стоматологической клиникой. Система охватывает полный цикл работы клиники: запись пациентов, ведение медицинских карт, расписание врачей, учёт оплат и базовую аналитику.

Система не является публичным сервисом — доступ только для сотрудников клиники и (в будущем) для пациентов по приглашению.

### Технический стек

| Слой | Технология |
|---|---|
| Backend | Python 3.11, Django 5.0.6, Django REST Framework 3.15 |
| Auth | `djangorestframework-simplejwt` — JWT (access 8ч, refresh 7д) |
| Database | PostgreSQL 14+ |
| Frontend | React 19, Vite 8, TypeScript ~6 |
| State | Zustand (auth/global), TanStack Query (server state) |
| HTTP | Axios с interceptor для auto-refresh токена |
| Стили | CSS Modules |
| Контейнеризация | Docker |

## 3. Модули системы

---

### М1 — Core (✅ Реализовано)

Базовые сущности и аутентификация. Является фундаментом для всех следующих модулей.

#### 3.1 Аутентификация

- Вход по email + пароль → JWT access + refresh токен
- JWT содержит: `role`, `email`, `full_name`
- Автоматическое обновление access-токена по refresh на 401
- Logout: удаление токенов из localStorage
- Защита роутов: `ProtectedRoute` с проверкой роли

**API:**
- `POST /api/v1/token/` — получение токенов
- `POST /api/v1/token/refresh/` — обновление access-токена

#### 3.2 Пациенты (Client)

Полная карточка пациента.

**Поля:** ФИО, ИИН (12 символов), телефон, пол, дата рождения, источник обращения, LTV, привязка к врачу.

**API:**
- `GET /api/v1/clients/` — список (с поиском по ФИО, ИИН, телефону)
- `POST /api/v1/clients/` — создание
- `GET /api/v1/clients/<id>/` — карточка
- `PATCH /api/v1/clients/<id>/` — редактирование
- `DELETE /api/v1/clients/<id>/` — удаление (только admin)

#### 3.3 Врачи и услуги

- **Doctors** — ФИО, специализация, услуги, описание
- **Assistant** — ФИО, привязка к услуге
- **Specialization** — название, стоимость
- **Service** — название, специализация

#### 3.4 Записи на приём (Record)

Основная транзакционная сущность.

**Поля:** пациент, врач, ассистент, услуга, зуб, стоимость, скидка, итог, дата/время начала и конца, кабинет, тип записи, тип оплаты, статус оплаты, статус приёма, заметки.

#### 3.5 Медицинская карта

- **MedicalNote** — анамнез, аллергии, заметки
- **ToothRecord** — статус каждого зуба (зубная формула)
- **TreatmentPlanItem** — план лечения по зубу
- **PatientFile** — файлы пациента (рентген, фото, документы)
  - Защищённая выгрузка через `/api/v1/medical/files/<pk>/download/` (JWT)
  - UUID-имена файлов (не угадываемые пути)

### М2 — Расписание (Calendar) 📅

Визуальный календарь записей. Основной рабочий инструмент администратора.

#### Требования

- Недельный и дневной вид
- Колонки по врачам или по кабинетам (переключатель)
- Цветовая кодировка по статусу записи
- Создание записи кликом по слоту
- Редактирование и перенос записи
- Проверка конфликтов: один врач не может быть записан дважды в пересекающееся время
- Фильтрация по врачу, кабинету, дате
- Индикатор загрузки врача (% занятости дня)

#### Новые API

- `GET /api/v1/records/calendar/?date_from=&date_to=&doctor=&chair=` — записи в диапазоне дат
- `GET /api/v1/records/slots/?doctor=&date=` — свободные слоты врача на дату
- `POST /api/v1/records/check-conflict/` — проверка конфликта перед созданием

#### Frontend

- Компонент `Calendar` на базе CSS Grid (без сторонних библиотек)
- Drag-and-drop для переноса записей (`@dnd-kit/core`)
- Страница `/admin/schedule` (уже существует, требует доработки)

#### Риски

- Логика конфликтов должна проверяться на backend, не только в UI
- Производительность: при большом числе записей нужна фильтрация по дате

### М3 — Финансы 💰

Учёт оплат, долгов и базовая финансовая отчётность.

#### Требования

- Фиксация оплаты по каждой записи (тип: наличные / карта / перевод)
- Статус оплаты: оплачено / частично / долг
- Список должников с суммой и датой последнего визита
- Ввод частичной оплаты с расчётом остатка
- История платежей по пациенту

#### Новые модели

```python
# payments/models.py
class Payment(models.Model):
    record = ForeignKey(Record, on_delete=CASCADE)
    amount = DecimalField(max_digits=10, decimal_places=2)
    payment_type = CharField  # cash / card / transfer
    paid_at = DateTimeField(auto_now_add=True)
    received_by = ForeignKey(User, on_delete=SET_NULL, null=True)
    notes = TextField(blank=True)
```

#### Новые API

- `GET /api/v1/payments/` — список платежей (фильтр по дате, врачу)
- `POST /api/v1/payments/` — зафиксировать оплату
- `GET /api/v1/payments/debts/` — список должников
- `GET /api/v1/payments/summary/?date_from=&date_to=` — итог за период

#### Frontend

- Вкладка "Оплата" в карточке записи
- Страница `/admin/finance` — сводка и список платежей
- Страница `/admin/debts` — должники
- Без интеграции с кассой (внутренний учёт)

### М4 — Дашборд и управление 🏥

#### 4.1 Дашборд

Главная страница после входа. KPI на сегодня / текущую неделю.

**Виджеты:**
- Приёмов сегодня / из них завершено
- Выручка за сегодня / за текущий месяц
- Новых пациентов за месяц
- Должники (количество и сумма)
- Ближайшие записи (следующие 3–5)
- Загрузка врачей (мини-таблица: врач → записей сегодня)

**API:**
- `GET /api/v1/dashboard/summary/` — агрегированные KPI

#### 4.2 Управление пользователями

Только для `admin`.

- Список сотрудников (врачи, администраторы)
- Создание нового пользователя с назначением роли
- Деактивация аккаунта (не удаление)
- Смена пароля пользователя администратором

**API:**
- `GET /api/v1/users/` — список
- `POST /api/v1/users/` — создание
- `PATCH /api/v1/users/<id>/` — редактирование
- `POST /api/v1/users/<id>/set-password/` — смена пароля
- `GET /api/v1/users/me/` — текущий пользователь

#### 4.3 Настройки клиники

- Название, адрес, телефон клиники
- Рабочие часы (по дням недели)
- Список кабинетов (ChairNum)
- Праздничные дни / выходные

**API:**
- `GET/PUT /api/v1/settings/clinic/`
- `GET/POST /api/v1/settings/working-hours/`

### М5 — Уведомления 🔔

SMS-напоминания о записях.

#### Требования

- Автоматическое напоминание за 24 часа до приёма
- Автоматическое напоминание за 2 часа до приёма
- Ручная отправка напоминания из карточки записи
- Шаблон сообщения настраивается администратором
- Лог отправок: статус (доставлено / ошибка), время, получатель

#### Архитектура

- Celery + Redis для периодических задач
- Таблица `SMSLog` для истории отправок
- SMS-провайдер: выбирается при реализации (SMSC.ru, SMS.kz или аналог)

#### API

- `POST /api/v1/notifications/send/<record_id>/` — ручная отправка
- `GET /api/v1/notifications/log/` — история отправок

#### Риски

- Зависимость от SMS-провайдера — выбрать до начала реализации
- Celery требует Redis — новая инфраструктурная зависимость

## 4. Нефункциональные требования

| Требование | Значение |
|---|---|
| Время ответа API | < 300 мс для списков, < 100 мс для простых GET |
| Покрытие тестами | ≥ 80% на каждый app |
| Аутентификация | JWT обязателен на всех endpoint'ах (кроме `/token/`) |
| Безопасность файлов | Только через `/download/` с JWT — без прямого доступа к media/ |
| Пагинация | Все списочные endpoint'ы: page size 20, настраиваемый |
| CORS | Только `http://localhost:5173` в dev; продакшн домен в prod |
| Секреты | `SECRET_KEY`, `DATABASE_URL` — только через переменные окружения |
| Логи | `logging.getLogger(__name__)` вместо `print()` |

## 5. Соглашения по коду

### Backend

- Type hints на всех функциях
- `ModelSerializer` для простого CRUD, `Serializer` для сложной валидации
- Тонкие вьюхи: вся логика в сериализаторе или отдельном сервисном слое
- Все модели с историей: поля `created_at` (auto_add) + `updated_at` (auto)
- Индексы на полях, используемых в `filter()` / `order_by()`
- Явный `permission_classes` в каждой вьюхе

### Frontend

- Axios только через `src/api/client.ts`
- Zustand для auth и глобального состояния
- TanStack Query для всего серверного состояния
- При добавлении endpoint: обновить и backend URL, и `frontend/src/api/`
- CSS Modules для стилей компонентов
