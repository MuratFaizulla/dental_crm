# Этап 2 — Медкарта пациента

**Дата:** 2026-05-24  
**Статус:** Согласовано  
**Зависит от:** Этап 1 (завершён)

---

## 1. Цель

Дать врачу и администратору полноценный инструмент для ведения медицинской карты пациента: интерактивная зубная формула, план лечения, хранение файлов (рентген, фото, документы).

---

## 2. Архитектура

### Подход
Новое Django-приложение `medical` внутри существующего проекта. Всё остаётся в одной базе данных, одном `docker-compose`, одном `manage.py`.

```
src/
  medical/
    models.py       — ToothRecord, TreatmentPlanItem, PatientFile, MedicalNote
    serializers.py
    views.py
    urls.py
    tests.py
    migrations/
```

### Доступ к медкарте
Два пути навигации:
- Список пациентов → карточка пациента → вкладка "Медкарта"
- Расписание → клик на запись → кнопка "Медкарта" → медкарта этого пациента

## 3. Модели данных

### 3.1 `ToothRecord` — статус зуба

| Поле | Тип | Описание |
|---|---|---|
| `patient` | FK → Client | Пациент |
| `tooth_number` | CharField | Номер зуба по FDI: "11"–"48" (взрослые), "51"–"85" (молочные) |
| `tooth_type` | CharField | `permanent` / `primary` |
| `status` | CharField | `healthy` / `treated` / `urgent` / `observation` / `missing` |
| `notes` | TextField | Комментарий врача (может быть пустым) |
| `updated_by` | FK → User | Кто последним изменил |
| `updated_at` | DateTimeField | Когда последним изменил |

Уникальность: `(patient, tooth_number)` — один статус на зуб на пациента.

По умолчанию зубы не создаются в БД — запись появляется только при первом изменении статуса (до этого считается `healthy`).

### 3.2 `TreatmentPlanItem` — пункт плана лечения

| Поле | Тип | Описание |
|---|---|---|
| `patient` | FK → Client | Пациент |
| `tooth_number` | CharField | Зуб |
| `diagnosis` | TextField | Диагноз |
| `treatment` | TextField | Рекомендуемое лечение |
| `service` | FK → Service (nullable) | Услуга из справочника |
| `due_date` | DateField (nullable) | Желаемый срок |
| `status` | CharField | `planned` / `in_progress` / `done` / `postponed` |
| `postpone_reason` | TextField (nullable) | Причина откладывания |
| `linked_record` | FK → Record (nullable) | Привязка к записи на приём |
| `doctor` | FK → User | Врач, создавший пункт |
| `created_at` | DateTimeField | — |

### 3.3 `PatientFile` — файл пациента

| Поле | Тип | Описание |
|---|---|---|
| `patient` | FK → Client | Пациент |
| `file` | FileField | Путь: `patients/{client_id}/{year}/{month}/{filename}` |
| `file_type` | CharField | `xray` / `photo` / `document` / `other` |
| `description` | CharField | Описание (необязательно) |
| `tooth_number` | CharField (nullable) | Привязка к конкретному зубу |
| `linked_record` | FK → Record (nullable) | Привязка к визиту |
| `uploaded_by` | FK → User | Кто загрузил |
| `uploaded_at` | DateTimeField | — |

Лимит: 20 МБ на файл. Форматы: jpg, png, webp, pdf, dcm.

### 3.4 `MedicalNote` — общие заметки

| Поле | Тип | Описание |
|---|---|---|
| `patient` | FK → Client (OneToOne) | Пациент |
| `anamnesis` | TextField | Анамнез |
| `allergies` | TextField | Аллергии |
| `notes` | TextField | Прочие заметки |
| `updated_by` | FK → User | Кто обновил |
| `updated_at` | DateTimeField | — |

## 4. API эндпоинты

Все маршруты подключаются через `src/api/urls.py` → `path('', include('medical.urls'))`.

**Автоматика:** когда `TreatmentPlanItem.status` меняется на `done` через PATCH, бэкенд автоматически обновляет соответствующий `ToothRecord.status` → `treated`. Фронтенд после этого перечитывает список зубов.

```
# Анамнез и заметки
GET    /api/v1/medical/{client_id}/note/             — получить заметки
PUT    /api/v1/medical/{client_id}/note/             — сохранить заметки (создать если нет)

# Зубная формула
GET    /api/v1/medical/{client_id}/teeth/            — все зубы с ненормальным статусом
PUT    /api/v1/medical/{client_id}/teeth/{number}/   — обновить статус одного зуба

# План лечения
GET    /api/v1/medical/{client_id}/plan/             — весь план лечения
POST   /api/v1/medical/{client_id}/plan/             — добавить пункт
PATCH  /api/v1/medical/{client_id}/plan/{id}/        — изменить статус / поля пункта
DELETE /api/v1/medical/{client_id}/plan/{id}/        — удалить пункт

# Файлы
GET    /api/v1/medical/{client_id}/files/            — список файлов
POST   /api/v1/medical/{client_id}/files/            — загрузить файл (multipart/form-data)
GET    /api/v1/medical/{client_id}/files/{id}/       — скачать / просмотреть файл
DELETE /api/v1/medical/{client_id}/files/{id}/       — удалить файл
```

### Права доступа
- **Администратор** — полный доступ ко всем пациентам
- **Врач** — только пациенты, у которых он указан как `doctor` в модели `Client`, или у которых есть хотя бы одна запись `Record` с этим врачом

## 5. Фронтенд

### 5.1 Маршрутизация (новые страницы)

```
/admin/patients/:id           — карточка пациента (новая)
/admin/patients/:id/medical   — медкарта: зубная формула + план лечения
/admin/patients/:id/files     — файлы пациента
```

Из расписания: клик на запись → появляется кнопка "Медкарта" → переход на `/admin/patients/:id/medical`.

Страница пациентов (уже есть) — добавить клик по строке → переход на `/admin/patients/:id`.

### 5.2 Карточка пациента `/admin/patients/:id`

Три вкладки:
1. **Информация** — ФИО, телефон, ИИН, дата рождения, врач, источник обращения, LTV, история визитов (последние 10 записей из `records`)
2. **Медкарта** — зубная формула + план лечения
3. **Файлы** — сетка файлов с загрузкой

### 5.3 Зубная формула (SVG)

Реализация: кастомный SVG в React. Внешние библиотеки не используются.

**Переключатель:** кнопки "Взрослые зубы" / "Молочные зубы" над формулой.

**Взрослые зубы (FDI):**
- Верхний ряд: 18 17 16 15 14 13 12 11 | 21 22 23 24 25 26 27 28
- Нижний ряд: 48 47 46 45 44 43 42 41 | 31 32 33 34 35 36 37 38

**Молочные зубы (FDI):**
- Верхний ряд: 55 54 53 52 51 | 61 62 63 64 65
- Нижний ряд: 85 84 83 82 81 | 71 72 73 74 75

**Цвета статусов:**

| Статус | Цвет | HEX |
|---|---|---|
| Здоров (по умолчанию) | Зелёный | `#52c41a` |
| Пролечен | Синий | `#1677ff` |
| Срочно | Красный | `#ff4d4f` |
| Наблюдение | Жёлтый | `#faad14` |
| Отсутствует | Серый | `#d9d9d9` |

**Клик по зубу** → открывается боковая панель справа:
- Заголовок: "Зуб №{number}"
- Текущий статус (цветной бейдж)
- Дропдаун: изменить статус
- Textarea: комментарий
- Кнопка "Сохранить"
- Кнопка "Добавить в план лечения" → открывает форму добавления пункта плана

### 5.4 План лечения

Список под зубной формулой. Каждый пункт:

```
[Зуб 36]  Кариес → Пломба композитная   Срок: 15.06.2026   [Запланировано ▼]  [Удалить]
```

- Статус — inline дропдаун: Запланировано / В процессе / Завершено / Отложено
- При выборе "Отложено" — появляется поле "Причина"
- При выборе "Завершено" — зуб в формуле автоматически меняет статус на "Пролечен"

### 5.5 Файлы `/admin/patients/:id/files`

Сетка карточек файлов. Кнопка "Загрузить файл" открывает форму:
- Drag & drop или выбор файла
- Тип: Рентген / Фото / Документ / Другое
- Описание (необязательно)
- Привязка к зубу (необязательно — номер из FDI)
- Привязка к визиту (необязательно — выпадающий список из истории записей)

Просмотр:
- Изображения и PDF — модальное окно в браузере
- Остальные форматы — скачивание

Каждая карточка файла показывает: тип, описание, привязанный зуб, дату загрузки, имя загрузившего.

## 6. Хранение файлов (Docker)

Файлы хранятся в `MEDIA_ROOT = BASE_DIR / 'media'`.  
В `docker-compose.yml` добавляется именованный volume чтобы файлы не терялись при рестарте:

```yaml
backend:
  volumes:
    - .:/DENTAL_CRM
    - media_files:/DENTAL_CRM/src/media

volumes:
  postgres_data:
  media_files:
```

В `settings.py`:
```python
MEDIA_URL = '/media/'
MEDIA_ROOT = BASE_DIR / 'media'
```

В `dental/urls.py` добавляется раздача медиафайлов в режиме DEBUG.

## 7. Тестирование

### Бэкенд (Django TestCase)
- Врач получает медкарту только своего пациента (403 для чужих)
- Администратор получает медкарту любого пациента
- Обновление статуса зуба сохраняется корректно
- Повторное обновление статуса того же зуба обновляет запись, не создаёт дубль
- Добавление пункта плана лечения
- Смена статуса пункта плана на `done`
- Загрузка файла: успешная загрузка, отклонение файла > 20 МБ
- Удаление файла удаляет запись и физический файл

### Фронтенд (ручная проверка)
- Из расписания → кнопка "Медкарта" → правильная страница
- Клик по зубу → боковая панель с данными
- Смена статуса зуба → цвет в SVG меняется без перезагрузки
- Добавление в план лечения → строка появляется в списке
- Статус "Завершено" → зуб становится синим
- Загрузка файла → карточка появляется в сетке
- Изображение открывается в модальном окне
